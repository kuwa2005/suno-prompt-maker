## 前提: 何を解くプロジェクトか

Suno Prompt Maker は、Suno AI 向け音楽プロンプトをブラウザ上で組み立てる **静的 Web アプリ**（HTML + vanilla JS + FTP デプロイ）です。

| 制約 | 内容 |
|------|------|
| インフラ | サーバーなし。API キー不要 |
| スタック | vanilla JS（IIFE）+ ES Modules の混在 |
| データ規模 | 38 カテゴリ、約 4,545 タグ |
| プライバシー | 入力テキストを外部に送信しない |

v1.1.0 で ternlight を統合し、次の 3 機能を追加しました。

1. **自動セッティング** — 自然言語 → カテゴリ % の一括設定
2. **セマンティック検索** — カテゴリ内タグの意味ベース絞り込み
3. **履歴検索** — localStorage 上の過去プロンプトのあいまい検索

本記事は「Suno の使い方」ではなく、**同種の制約下でクライアントサイド埋め込み検索を載せるときの設計判断**を中心に書きます。

## アーキテクチャの全体像

### 3 層構成

```
[データ層]     data.js — CATEGORIES + PROMPT_DATA
[インデックス層] tag-index.js — ベクトルキャッシュ + searchTags()
[エンジン層]   ternlight-engine.js — embed / cosineSim の抽象化
[アプリ層]     auto-setter / semantic-search / history-search
[既存 UI]     app.js (IIFE) — window 経由で連携
```

ternlight は **「類似度スコアを返す部品」** に留め、カテゴリへの集約・閾値・除外ルールはすべてアプリ層の責務です。この分離が、フォールバック切り替えとドメインルール追加を両立させています。

### 処理パイプライン

```
ユーザー入力
  → embedText()        # テキスト → ベクトル（またはトークン配列）
  → cosineSimilarity() # 0〜1 のスコア
  → アプリ固有ルール   # 閾値・正規化・カテゴリ除外
  → UI 反映
```

**設計上の判断:** ternlight に「何を選ぶか」を任せない。音楽プロンプトでは「ジャンルはランダム要素として残す」といったドメイン知識があり、これは `auto-setter.js` 側で `EXCLUDED_CATEGORIES` として明示しています。

## 既存 vanilla JS への組み込みパターン

### 問題: IIFE と ES Modules の共存

`app.js` は従来どおり IIFE 形式。AI 機能は ES Modules で追加しています。全面リファクタは避け、**window オブジェクトを境界**にしました。

```html
<script src="data.js"></script>
<script src="app.js"></script>
<script type="module" src="ternlight-engine.js"></script>
<script type="module">
  import { initTernlight } from './ternlight-engine.js';
  import { buildTagIndex } from './tag-index.js';
  // ...

  async function initializeAI() {
    await initTernlight();
    await buildTagIndex();
    // 検索 UI 追加・イベント登録
  }
  initializeAI();
</script>
```

AI モジュール側から `window.weightSliders` や `window.updateWeightLabels` を参照します。レガシーコードを触らずに機能追加できる反面、**グローバル依存はテストとリファクタの負債**になります。Vite 移行時にここを整理する予定です。

### ファイル責務の切り方

| ファイル | 責務 |
|----------|------|
| `ternlight-engine.js` | エンジン初期化、`embedText` / `cosineSimilarity` ラッパー、日本語対応 |
| `tag-index.js` | 全タグのベクトルキャッシュ、`searchTags()` |
| `semantic-search.js` | カテゴリ別検索 UI（DOM 操作） |
| `auto-setter.js` | 自然言語 → スライダー % の変換ロジック |
| `history-search.js` | 履歴の遅延インデックス構築と検索 |
| `data.js` | `CATEGORIES` + `PROMPT_DATA`（単一の真実の源） |

**ポイント:** 埋め込みエンジンとドメインロジックを分離することで、WASM ↔ フォールバックの差し替えが `ternlight-engine.js` 1 ファイルに閉じます。

## エンジンラッパー: API 形状の固定

### 公開インターフェース

```javascript
initTernlight()           // 非同期初期化
embedText(text)           // テキスト → ベクトル
cosineSimilarity(a, b)    // 類似度 0〜1
isReady()                 // 初期化完了
isFallbackMode()          // フォールバック稼働中か
```

上層モジュールは **WASM 版かフォールバック版かを意識しない** 設計です。

### WASM モード（計画）

```javascript
const module = await import('https://cdn.jsdelivr.net/npm/@ternlight/base@latest/dist/index.mjs');
embedFn = module.embed;
cosineSimFn = module.cosineSim;
```

### フォールバックモード（現行本番）

WASM の読み込みにはバンドラーが必要で、FTP で HTML/JS をそのまま配信する現行デプロイでは CDN 経由の WASM ロードが使えません。そのため内部実装をキーワード一致に差し替えています。

```javascript
// 英語 → 単語トークン配列
// 日本語 → JP_FALLBACK_MAP で英語トークンへ変換
embedFn = (text) => { /* トークン化 */ };
cosineSimFn = (a, b) => { /* Jaccard 風の集合類似度 */ };
fallbackMode = true;
```

### なぜフォールバックを先に本番投入したか

| メリット | 説明 |
|----------|------|
| 上層コードを書き切れる | `tag-index.js` 等は本番投入前に完成 |
| 日本語辞書を先行実装 | `JP_FALLBACK_MAP` / `jpCategoryMap` は WASM 移行後も有効 |
| 障害耐性 | CDN / WASM ロード失敗時もアプリ全体は動く |
| UX の早期検証 | 閾値・除外ルールをフォールバックで調整済み |

**トレードオフ:** フォールバックは文脈理解が弱く、「カフェの BGM」のような複合表現は個別キーワード（カフェ、チル）への依存が強くなります。WASM 移行でここが最も改善が見込める部分です。

## データ設計: 2 層構造とインデックス

### CATEGORIES + PROMPT_DATA

カテゴリのメタ情報とタグ候補を分離しています。

```javascript
// CATEGORIES — UI・スライダー用メタデータ
const CATEGORIES = {
  genre: {
    name: 'ジャンル',
    label: 'ジャンル',
    dataKey: 'genres',      // ← PROMPT_DATA のキーと対応
    defaultWeight: 0,
  },
  // ... 計 38 カテゴリ
};

// PROMPT_DATA — タグ文字列の配列
const PROMPT_DATA = {
  genres: ['Pop', 'Rock', 'City Pop', 'Synthwave', /* ... */],
  styles: ['smooth', 'energetic', 'chill', /* ... */],
  // ...
};
```

**運用上の落とし穴:** `dataKey` の不一致は AI 機能のサイレント障害になります。カテゴリ追加はこの 2 箇所の編集だけで UI・プロンプト生成・AI インデックスすべてに反映される設計ですが、キー名の整合性チェックは必須です。

### タグインデックス構築

```javascript
// tag-index.js — ページ読み込み時に全タグをベクトル化
for (const [catId, cat] of Object.entries(CATEGORIES)) {
  const items = PROMPT_DATA[cat.dataKey] || [];
  for (const item of items) {
    const vector = embedText(item);
    index.push({ category: catId, tag: item, vector: Array.from(vector) });
  }
}
```

**規模感（2026 年 7 月時点）**

- カテゴリ: 38
- タグ総数: 約 4,545
- 初回ベクトル化: 設計目標 < 3 秒（WASM 時）
- フォールバック時の検索: 体感 10ms 前後（トークン + 集合演算）

### sessionStorage キャッシュ

```javascript
const CACHE_KEY = 'suno_tag_vectors';
// ヒット時は JSON.parse して即復元、ミス時は buildTagIndex() で再構築
```

WASM 本番運用時は `Float32Array` を JSON 化してキャッシュする点に注意が必要です。`data.js` 更新時はキャッシュキーのバージョン管理、またはビルドハッシュによる無効化を検討してください。

### 検索 API

```javascript
export function searchTags(query, topK = 10) {
  const queryVector = embedText(query);
  const results = tagIndex.map(item => ({
    ...item,
    score: cosineSimilarity(queryVector, item.vector),
  }));
  results.sort((a, b) => b.score - a.score);
  return results.slice(0, topK);
}
```

クエリ 1 回 = `embedText` 1 回 + 全タグとの類似度計算 1 回。O(n) ですが 4,500 件規模ならクライアントサイドで十分です。タグが 10 万件規模に膨らむ場合は、インデックス構造の見直し（近似最近傍探索など）が必要になります。

## 日本語入力への 3 段構え

タグデータは英語表記が中心（`'City Pop'`, `'Chillout'` 等）。日本語の自由記述を扱うため、エンジン層に以下を実装しています。

1. **JP_FALLBACK_MAP / jpCategoryMap** — 日本語キーワード → 英語トークン・カテゴリ ID の直接マッピング
2. **categoryMap（英語キーワード辞書）** — 英語テキストからカテゴリごとの重み付きスコアを算出
3. **Chrome Translator API**（Chrome 138+、オプション）— 利用可能なら日本語 → 英語翻訳後にキーワード抽出

Translator API はユーザージェスチャー時に初期化するなど、ブラウザ API の制約に合わせた実装です。フォールバックモードでは `isFallbackMode()` が true のとき、`auto-setter.js` 側で `extractCategoriesWithTranslation()` による日本語向け追加ロジックが走ります。

## 3 機能の実装詳細

### 1. 自動セッティング（auto-setter.js）

**入力:** 「お洒落なカフェで流れるチルいアコースティック曲」  
**出力:** ジャンル以外のスライダー % を一括設定

```javascript
const EXCLUDED_CATEGORIES = ['genre'];

// フォールバック: 閾値 0.05 / WASM 本番: 閾値 0.3
if (result.score > threshold) {
  categoryScores[result.category] += result.score;
}
```

**処理フロー**

1. 入力テキストを解析し、カテゴリごとのスコアを集計
2. フォールバック時は `extractCategoriesWithTranslation()` + `searchTags()` の二段構え
3. `EXCLUDED_CATEGORIES` でジャンルを除外
4. スコアを正規化して % に変換、全スライダーを 0 リセット後に反映

**設計の要点:** ternlight の類似度は「タグ文字列同士の近さ」であり、「カテゴリの %」というアプリ固有の概念ではありません。カテゴリ単位への集約・正規化・除外はアプリ層の責務です。

### 2. セマンティック検索（semantic-search.js）

カテゴリ内のタグ DOM に対し、検索クエリとの類似度で視覚的に並べ替えます。

```javascript
const results = searchTags(query, 50).filter(r => r.category === categoryId);
// マッチ: opacity 1 / 非マッチ: opacity 0.4 + order で後退
```

`'laid-back'` と `'relaxed'` のような表現揺れを、完全一致検索ではなくトークン / ベクトル類似度でカバーします。初期化時に `addSearchBoxes()` が全 38 カテゴリ分の検索窓を動的生成します。

### 3. 履歴検索（history-search.js）

```javascript
vector: embedText(entry.promptLine + ' ' + (entry.comment || ''))
```

- 保存先: localStorage（最大 50 件）
- インデックス: 初回検索時に `buildHistoryIndex()` で遅延構築、メモリ上に保持
- 閾値: 類似度 0.3 未満は除外
- 無効化: 履歴の追加・削除時は `clearHistoryIndex()` で破棄し、次回検索時に再構築（`app.js` の履歴操作と連携）

## 移植するときのチェックリスト

同様の構成を別プロジェクトに載せる場合の要点です。

### 必須

- [ ] 埋め込みエンジンをラッパーで抽象化し、上層は `embed` / `cosineSim` だけに依存させる
- [ ] ドメインデータ（タグ一覧）とメタデータ（カテゴリ定義）を分離する
- [ ] 類似度スコア → アプリ固有の出力（%, ランキング, フィルタ）の変換ルールをアプリ層に閉じ込める
- [ ] インデックスのキャッシュ戦略（sessionStorage + バージョンキー）を決める

### 日本語対応がある場合

- [ ] タグ言語と入力言語のギャップをどう埋めるか（辞書 / 翻訳 API / 両方）を先に決める
- [ ] フォールバックと WASM で閾値が異なる場合、`isFallbackMode()` で分岐する

### WASM 本番化

- [ ] バンドラー（Vite 等）で WASM アセットを同梱する
- [ ] 初回ロード 7MB の UX（ローディング表示）を設計する
- [ ] フォールバック自動切り替えのエラーハンドリングを残す

## 今後: Vite による WASM 本番化

リポジトリ内の移行計画（`docs/compose/plans/2026-07-08-vite-bundler.md`）に記載のとおり、ternlight WASM の本格利用にはバンドラー導入が前提です。

```bash
npm install @ternlight/base
npm run build   # dist/ に WASM 同梱
./deploy-ftp.sh # dist を FTP デプロイ
```

**移行時の変更点**

| 項目 | フォールバック（現行） | WASM 本番（予定） |
|------|------------------------|-------------------|
| 類似度閾値（自動セッティング） | 0.05 | 0.3 |
| 日本語処理 | `extractCategoriesWithTranslation()` 必須 | ベクトル類似度でカバー可能 |
| 初回ロード | 軽量 | WASM 7MB + ローディング UI |
| デプロイ | HTML/JS を FTP 直送 | `dist/` をビルド後にデプロイ |

## まとめ: 設計パターンとしての ternlight 活用

1. **エンジンラッパー** — `ternlight-engine.js` で API を固定し、WASM / フォールバックを透過的に切り替える
2. **データ駆動インデックス** — `CATEGORIES` + `PROMPT_DATA` から `buildTagIndex()` で自動構築、`sessionStorage` でキャッシュ
3. **ルールベース後処理** — 類似度スコアをそのまま UI に渡さず、閾値・正規化・除外ルールをアプリ層で適用

ternlight は「魔法の自動設定エンジン」ではなく、**大量の候補と自然言語の間に意味的な橋を架ける部品** です。静的サイトのまま AI っぽい UX を足したいプロジェクトでは、この 3 層構成がそのまま参考になるはずです。
