## はじめに

Suno Prompt Maker は、Suno AI 向けの音楽プロンプトをブラウザ上で組み立てる静的 Web アプリです。38 カテゴリ・4,500 件超のタグ候補から要素を選び、出現割合（%）を調整してプロンプトを生成します。

**GitHub:** https://github.com/kuwa2005/suno-prompt-maker  
**本番デモ:** https://debugprint.com/suno-prompt-maker/

v1.1.0 以降、**ternlight** を組み込み、次の 3 つの AI 機能を追加しました。

1. **自動セッティング** — 自然言語入力からカテゴリの % を自動調整
2. **セマンティック検索** — 各カテゴリのタグ候補を意味ベースで絞り込み
3. **履歴検索** — 保存済みプロンプトを意味ベースで検索

いずれも **サーバー不要・クライアントサイド完結** という制約のもと設計されています。本記事では、組み込み方・データの用意・実際の活用方法を、本プロジェクトの実装に沿って解説します。

※ 本番アプリの画面キャプチャをここに 1 枚挿入すると、読者のイメージが掴みやすくなります。

## ternlight とは何を担うか

ternlight（`@ternlight/base`）は、テキストをベクトル（埋め込み）に変換し、コサイン類似度で意味的な近さを計算するライブラリです。WASM ベースでブラウザ上で動作し、外部 API への送信なしにセマンティック検索が可能です。

本プロジェクトでは、ternlight を **「意味の距離を測るエンジン」** として位置づけ、UI 側のルール（閾値・正規化・除外カテゴリなど）と組み合わせる **ハイブリッド構成** を採用しました。

**処理の流れ（概要）**

```
ユーザー入力
    ↓
embed() でベクトル化
    ↓
cosineSim() で類似度スコア
    ↓
ルールベース変換（閾値・正規化・除外）
    ↓
UI 反映（スライダー / タグ表示 / 履歴一覧）
```

ternlight 単体で「何を選ぶか」を決めるのではなく、**類似度スコアをアプリ固有のルールに渡す** 設計にすることで、音楽プロンプトというドメイン知識（例: ジャンルはランダム要素として残す）を反映しやすくしています。

## 組み込み方

### 技術スタックとファイル構成

主要ファイルと役割は次のとおりです。

- **ternlight-engine.js** — ternlight の初期化、`embedText()` / `cosineSimilarity()` のラッパー
- **tag-index.js** — 全タグのベクトルキャッシュ構築、`searchTags()`
- **semantic-search.js** — カテゴリ別タグ絞り込み UI
- **auto-setter.js** — 自然言語 → スライダー % の自動算出
- **history-search.js** — 履歴のあいまい検索
- **data.js** — カテゴリ定義（`CATEGORIES`）とタグデータ（`PROMPT_DATA`）
- **index.html** — ES Modules による AI 機能の初期化・イベント配線

`package.json` には `@ternlight/base` が dependencies として登録されています。当初は CDN から WASM を読み込む構成を想定していましたが、**WASM の読み込みにはバンドラーが必要** なため、現行の静的デプロイ（FTP で HTML/JS をそのまま配信）では **フォールバックモード** で動作しています（後述）。

### HTML への組み込み

`index.html` では、従来のグローバルスクリプト（`data.js` → `app.js`）に加え、ES Modules で AI 関連モジュールを読み込みます。

```html
<script src="data.js"></script>
<script src="app.js"></script>
<script type="module" src="ternlight-engine.js"></script>
<script type="module">
  import { initTernlight, translateMainPrompt } from './ternlight-engine.js';
  import { buildTagIndex } from './tag-index.js';
  import { createSearchBox, filterTagElements } from './semantic-search.js';
  import { autoSetWeights } from './auto-setter.js';
  import { searchHistory } from './history-search.js';

  async function initializeAI() {
    await initTernlight();
    await buildTagIndex();
    // 検索 UI 追加・イベントハンドラ登録
  }
  initializeAI();
</script>
```

`app.js` は従来どおり IIFE 形式のままです。AI 機能側から `window.weightSliders` や `window.updateWeightLabels` などを参照するため、**既存コードを大きく書き換えずにモジュールを足す** 形にしています。

### エンジンラッパー（ternlight-engine.js）

設計上のインターフェースは次のとおりです。

- **initTernlight()** — エンジン初期化（非同期）
- **embedText(text)** — テキスト → ベクトル（またはトークン配列）
- **cosineSimilarity(vecA, vecB)** — 類似度 0〜1
- **isReady() / isFallbackMode()** — 状態確認
- **translateMainPrompt(text)** — メインプロンプトの日本語→英語翻訳（Chrome Translator API）
- **extractCategoriesWithTranslation(text)** — 日本語入力からカテゴリスコアを抽出

**本番 WASM モード（計画）** では、次のように CDN / npm から `@ternlight/base` を読み込み、`embed` と `cosineSim` をそのまま使う想定でした。

```javascript
const module = await import('https://cdn.jsdelivr.net/npm/@ternlight/base@latest/dist/index.mjs');
embedFn = module.embed;
cosineSimFn = module.cosineSim;
```

**現行フォールバックモード** では、同じ API 形状を保ちつつ、内部実装をキーワード一致に差し替えています。

```javascript
// 英語テキスト → 単語トークン
// 日本語テキスト → JP_FALLBACK_MAP から英語トークンへ変換
embedFn = (text) => { /* トークン化 */ };
cosineSimFn = (a, b) => { /* Jaccard 風の集合類似度 */ };
fallbackMode = true;
```

これにより、上層の `tag-index.js` や `auto-setter.js` は **WASM 版とフォールバック版を意識せず同じコード** で動作します。`isFallbackMode()` が true のときだけ、自動セッティング側で日本語向けの追加ロジック（後述）が走ります。

### 日本語入力への対応

タグデータは英語表記が中心（例: `'City Pop'`, `'Chillout'`）です。日本語の自由記述（「お洒落なカフェで流れるチルい曲」など）を扱うため、次の 3 段構えを実装しています。

1. **JP_FALLBACK_MAP / jpCategoryMap** — よく使う日本語キーワードを英語トークン・カテゴリ ID に直接マッピング
2. **categoryMap（英語キーワード辞書）** — 英語テキストからカテゴリごとの重み付きスコアを算出
3. **Chrome Translator API**（Chrome 138+、オプション）— 利用可能な場合、日本語を英語に翻訳してからキーワード抽出

Translator API はユーザージェスチャー時に初期化するなど、ブラウザ API の制約に合わせた実装になっています。

## データの用意の仕方

### 2 層構造: CATEGORIES と PROMPT_DATA

`data.js` では、カテゴリのメタ情報とタグ候補を分離しています。

**CATEGORIES** — スライダー・UI 表示用のメタデータ

```javascript
const CATEGORIES = {
  genre: {
    name: 'ジャンル',
    label: 'ジャンル',
    desc: '',
    dataKey: 'genres',      // PROMPT_DATA のキーと対応
    defaultWeight: 0,
  },
  style: {
    name: 'スタイル',
    label: 'スタイル',
    desc: '曲の雰囲気・表現スタイル',
    dataKey: 'styles',
    defaultWeight: 0,
  },
  // ... 計 38 カテゴリ
};
```

**PROMPT_DATA** — 各カテゴリのタグ文字列配列

```javascript
const PROMPT_DATA = {
  genres: ['Pop', 'Rock', 'City Pop', 'Synthwave', /* ... */],
  styles: ['smooth', 'energetic', 'chill', /* ... */],
  moods: ['happy', 'melancholic', 'dreamy', /* ... */],
  // ...
};
```

カテゴリを追加するときは、この 2 箇所を編集するだけで、プロンプト生成・UI・AI インデックスのすべてに反映されます。`dataKey` の不一致は AI 機能のサイレント障害につながるため、**キー名の一致** が運用上の重要ポイントです。

### タグインデックスの構築（tag-index.js）

ページ読み込み時、`buildTagIndex()` が全タグを走査し、1 タグ 1 ベクトルのインデックスを構築します。

```javascript
for (const [catId, cat] of Object.entries(CATEGORIES)) {
  const items = PROMPT_DATA[cat.dataKey] || [];
  for (const item of items) {
    const vector = embedText(item);
    index.push({ category: catId, tag: item, vector: Array.from(vector) });
  }
}
```

**規模感（2026 年 7 月時点）**

- カテゴリ数: 38
- タグ総数: 約 4,545
- 初回ベクトル化: 設計目標 < 3 秒（WASM 時）

**sessionStorage キャッシュ** により、同一セッション内の再読み込みではベクトル化をスキップします。

```javascript
const CACHE_KEY = 'suno_tag_vectors';
// キャッシュヒット時は JSON.parse して即復元
```

WASM 本番運用時は Float32Array を JSON 化してキャッシュする点に注意が必要です。データ更新時はキャッシュキーのバージョン管理や、ビルドハッシュによる無効化を検討する価値があります。

### 検索 API

```javascript
export function searchTags(query, topK = 10) {
  const queryVector = embedText(query);
  const results = tagIndex.map(item => ({
    ...item,
    score: cosineSimilarity(queryVector, item.vector),
  }));
  results.sort((a, b) => b.score - a.score);
  return results.slice(0, topK);
}
```

クエリ 1 回につき `embedText` 1 回 + 全タグとの類似度計算 1 回。4,500 件規模であれば、フォールバック（トークン + 集合演算）でも体感 10ms 前後を狙える設計です。

## 3 つの活用方法

### 1. 自動セッティング（auto-setter.js）

**ユースケース:** メインプロンプトに「お洒落なカフェで流れるチルいアコースティック曲」と書き、「メインプロンプトから反映」でスライダーを一括設定。

**処理フロー**

1. 入力テキストを解析し、カテゴリごとのスコアを集計
2. フォールバック時は `extractCategoriesWithTranslation()` で日本語→カテゴリを直接推定し、さらに `searchTags()` でタグ類似度を加算
3. **ジャンル（genre）は意図的に除外** — ジャンルはランダム要素として残し、雰囲気・楽器・ムードなどを AI が調整
4. スコアを正規化して % に変換（合計 100% 相当）、全スライダーを 0 にリセットしてから反映

```javascript
const EXCLUDED_CATEGORIES = ['genre'];

// フォールバック: 類似度閾値 0.05
// WASM 本番: 類似度閾値 0.3
if (result.score > threshold) {
  categoryScores[result.category] += result.score;
}
```

**設計上のポイント:** ternlight の類似度は「タグ文字列同士の近さ」であり、「カテゴリの %」というアプリ固有の概念ではありません。カテゴリ単位への集約・正規化・除外ルールは **アプリ側の責務** として `auto-setter.js` に閉じ込めています。

※ 「メインプロンプトから反映」実行前後のスライダー画面をここに挿入すると効果的です。

### 2. セマンティック検索（semantic-search.js）

**ユースケース:** 「要素候補詳細」を開き、各カテゴリの「🔍 候補を絞り込む」に `warm` や `reverb` と入力。意味が近いタグが前面に、遠いタグは半透明表示。

```javascript
const results = searchTags(query, 50).filter(r => r.category === categoryId);
```

UI 側では DOM の `.tag` 要素に対し、マッチしたタグは `opacity: 1`、それ以外は `opacity: 0.4` と `order` で視覚的に後退させます。完全一致検索では見つからない `'laid-back'` と `'relaxed'` のような表現揺れを、トークン / ベクトル類似度でカバーします。

初期化時に `addSearchBoxes()` が全カテゴリ分の検索窓を動的生成します。

### 3. 履歴検索（history-search.js）

**ユースケース:** 履歴タブの「🔍 履歴を検索（意味ベース）」で、過去に作ったプロンプトをキーワードではなく意味で探す。

```javascript
vector: embedText(entry.promptLine + ' ' + (entry.comment || ''))
```

履歴は localStorage に最大 50 件。初回検索時に `buildHistoryIndex()` でベクトル化し、メモリ上に保持します。類似度 0.3 未満は除外します。

履歴の追加・削除時は `clearHistoryIndex()` でインデックスを破棄し、次回検索時に再構築する必要があります（`app.js` の履歴操作と連携）。

## 実際にどう活用できたか

### ユーザー体験の変化

**従来**

- 38 個のスライダーを手動調整
- タグ一覧を目視でスクロール
- 履歴は時系列のみ

**ternlight 導入後**

- 自然言語 1 文でおおまかな配分を自動設定
- カテゴリ内セマンティック検索で候補を絞り込み
- 「夏っぽい」「シティポップ風」など意味で過去作を検索

いずれも **外部 API キー不要・プライバシー配慮**（入力テキストはブラウザ外に送信しない）という静的サイトの強みを維持したまま実現しています。

### 開発上のメリット

1. **関心の分離** — 埋め込みエンジン（`ternlight-engine.js`）とドメインロジック（`auto-setter.js`）を分離でき、フォールバック切り替えが容易
2. **データ駆動** — `data.js` を編集するだけで AI インデックス対象が増える
3. **段階的導入** — まずフォールバックで UX を完成させ、Vite 導入後に WASM 本番へ移行する計画が立てやすい

### フォールバック運用で得た知見

現行本番は WASM ではなくキーワードベースのフォールバックですが、それでも以下の価値がありました。

- **API 形状の固定** — 上層モジュールを書き切れる
- **日本語対応の先行実装** — `jpCategoryMap` などドメイン辞書は WASM 移行後も有効
- **オフライン耐性** — CDN / WASM ロード失敗時も基本機能が動く

一方、フォールバックでは「カフェの BGM」のような **文脈全体の理解** は弱く、個別キーワード（カフェ、チル）への依存が強くなります。本番 WASM への移行で、ここが最も改善が見込める部分です。

## 今後の展望: Vite による本番 WASM 化

リポジトリ内の Vite 移行計画（`docs/compose/plans/2026-07-08-vite-bundler.md`）に記載のとおり、ternlight WASM を本格利用するには **Vite 等のバンドラー導入** が前提です。

```
npm install @ternlight/base
npm run build   # dist/ に WASM 同梱
./deploy-ftp.sh # dist を FTP デプロイ
```

**移行時のチェックリスト**

- `ternlight-engine.js` の `initTernlight()` で `fallbackMode = false` かつ npm 版 `embed` / `cosineSim` を使用
- WASM アセットの public 配置または Vite の `assetsInclude` 設定
- `sessionStorage` キャッシュのスキーマバージョン管理
- 自動セッティングの閾値（0.3）をフォールバック用（0.05）から統一
- 初回ロード 7MB WASM の UX（ローディング表示など）

## エラーハンドリングとテスト

### フォールバック

WASM 初期化失敗時は、キーワード一致ベースの `embed` / `cosineSim` に自動切り替え。AI 機能は劣化しますが、アプリ全体は利用可能です。

### テスト

- **test-auto-setter.js** — 日本語入力 → 期待カテゴリの Node.js 単体テスト
- **test-auto-setter.html** — ブラウザ上でのカテゴリ抽出確認
- **test-integration.html** — エンジン初期化・抽出の結合確認

カテゴリマッピングを変更した際は、`test-auto-setter.js` の期待値も合わせて更新する運用が有効です。

## まとめ

Suno Prompt Maker における ternlight の活用は、次の 3 点に集約されます。

1. **組み込み** — `ternlight-engine.js` で API をラップし、ES Modules で既存 vanilla JS アプリに段階的に接続
2. **データ** — `CATEGORIES` + `PROMPT_DATA` の 2 層構造からタグインデックスを自動構築し、sessionStorage でキャッシュ
3. **活用** — 類似度スコアを自動セッティング・タグ検索・履歴検索の 3 機能に配分し、ドメインルール（ジャンル除外、閾値、日本語辞書）と組み合わせ

ternlight は「魔法の自動設定」ではなく、**大量のタグ候補と自然言語の間に意味的な橋を架ける部品** として機能しています。静的サイトのまま AI っぽい UX を足したいプロジェクトでは、エンジンラッパー + データ駆動インデックス + ルールベース後処理、という構成が参考になるはずです。
